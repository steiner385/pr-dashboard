# Screen Inventory & Use-Case Analysis

> **Purpose of this document.** A complete inventory of every screen/surface in the
> pr-dashboard front-end, paired with a *brainstorm* of the intended and possible
> use cases (user stories) each screen could serve. Use cases are written
> **aspirationally** — they deliberately ignore what is or isn't implemented today, so
> the list doubles as a backlog of opportunity and a check against the original design
> intent.
>
> **Method.** The inventory was produced by reading the routing model
> (`frontend/src/shell/sections.ts`) and the section/overlay components directly. The
> use cases were generated by role-playing **five user personas** against every screen
> — Release Manager, PR Author (developer), Platform / DevEx Engineer, Engineering
> Manager, and SRE / On-call. Each persona walked all surfaces and produced stories
> from their own goals. Stories below are de-duplicated and attributed by persona tag:
> `[RM]` Release Manager · `[DEV]` PR Author · `[PLAT]` Platform Engineer ·
> `[EM]` Eng Manager · `[SRE]` On-call.

---

## 1. How the app is organized

The product is a **unified workspace** with **five verb-named sections**, each carrying
an interaction *mode* that was derived from a persona review (see the header comment in
`sections.ts`). The mode drives how much detail is shown by default vs. on demand.

| Section | Mode | One-line intent |
|---|---|---|
| **Health** | `monitor` | Is delivery healthy right now? |
| **Pipeline** | `monitor` | Every open PR and where it is in the pipeline. |
| **Diagnose** | `drill` | Why is this PR stuck? |
| **Model & Edit** | `act` | Inspect what gates a merge, find waste, and shape the pipeline. |
| **Insights** | `read` | Cost, queue, runners, flake, lead time, budgets, outcomes — the full analytics. |

Five legacy tabs were consolidated into these (hash aliases preserved):
`tune` + `metrics` → **Insights**; `model` + `optimize` + `build` → **Model & Edit**.

Around the five sections sits a **shell** (header "spine", left rail, command palette,
overlays) plus two alternate delivery modes — a **standalone classic app** and an
**embeddable component** — and a **kiosk/wall-display** mode. These are inventoried as
group 6.

### Surface map at a glance

```
Workspace shell
├─ Header spine: brand · pipeline switcher · liveness · self-health dot · ⌘K · legend(?) · 🔔 · settings(⚙)
├─ Left rail: Health · Pipeline · Diagnose · Model & Edit · Insights
└─ Main content (active section)

Sections
├─ Health        → Delivery Spine (7 lanes) · Fleet rollup · Flakiest leaderboard · Forecast banner
├─ Pipeline      → Status tiles · per-repo sections · Queue train · PrRow (CheckGantt, Waterfall, MetroTrack)
├─ Diagnose      → Remediation/incident/cluster banners · PR list · blocker + CheckGantt
├─ Model & Edit  → tabs: Inspect (matrix + drawer) · Optimize (findings/levers) · Edit (DAG canvas)
└─ Insights      → tabs: Analytics (5 sub-sections) · Tuning & Policy (budgets/policy/outcomes/changelog)

Overlays         → Command Palette · Settings Panel · Legend Panel · Protection Drawer · Forecast Banner · Status Strip
Alt delivery     → Standalone classic app · Embeddable component · Kiosk mode
```

## 2. Health  *(mode: monitor)*

### What it is
The "is delivery healthy right now?" screen. A live **Delivery Spine** of seven lanes —
PR CI, Merge Queue, Main, Deploy, Scheduled, Failures/Flake, Cost — each showing a
green/amber/red/blind/idle status with a one-line summary; expandable into a per-lane
detail panel. Above it sits a compact **HealthHeader** rollup (one pill + clickable lane
chips). Below it, a **Fleet rollup** lists every watched repo ranked by attention
(down → attention → healthy) with a verdict reason on hover, a **Flakiest-pipelines
leaderboard** (top 5), and a conditional **cost Forecast banner** ("budget runs out in
~N days"). Liveness/self-health indicators back the "green means green" guarantee.

### Intended & possible use cases

- `[RM]` Glance at all 7 lanes and know in 5 seconds whether it's safe to open the merge
  window, without digging.
- `[SRE]` Have lanes flip red within seconds of a real regression (event-driven, not a
  5-min poll) so I learn about a red main before the first DM.
- `[SRE]` Trust the signal: a liveness state (live/stale/reconnecting) + self-health dot
  so a dead feed never masquerades as a healthy fleet.
- `[SRE/PLAT]` See *amber = degrading-not-yet-breached* with the threshold shown, so I
  can pre-empt a red instead of only reacting to it.
- `[RM/PLAT]` Have a red lane name the downstream lanes it's blocking (Main red → Deploy
  can't promote) so I understand blast radius before acting.
- `[EM]` Roll the whole fleet into one weighted health score (by team / business
  criticality) for a 5-second morning read.
- `[EM/SRE]` Show a delta-vs-last-week (or "since my last visit") trend arrow per lane so
  a *degrading green* is distinguishable from a *stable green*.
- `[DEV]` Answer "is it me or is it everyone?" in one line so I don't debug an outage
  that isn't mine before blaming my own diff.
- `[PLAT/EM]` Quantify the flakiest leaderboard in wasted engineer-hours and re-run
  dollars so "annoying" becomes a fundable line item.
- `[RM/PLAT]` Click any red/amber lane to jump straight into Diagnose (or Insights for
  cost) pre-scoped to that lane — detection flows into triage in one hop.
- `[EM]` Snapshot the screen as a dated image/PDF for the weekly leadership update.
- `[RM]` Surface a cost-forecast warning when a release-day re-run storm threatens the
  budget, so I can throttle retries before finance notices.

**Home base for:** SRE/On-call (the "are we shipping?" source of truth) and Release
Manager (pre-flight + standby). Also the natural kiosk default.

## 3. Pipeline  *(mode: monitor)*

### What it is
Every open PR and exactly where it sits in the pipeline (CI → Queue → Merged → QA →
Prod). A top **Status Strip** of five filter tiles (CI running / In queue / Awaiting
prod / Failed / Ready·other) with a **failed-count delta badge** (5-min spike
detection). Per repo: a **Queue "metro train"** with building/waiting cars and p50/p90
ETAs, deploy backlog/cohort notes, and a list of **PrRow**s. Each PrRow expands to a
**MetroTrack** stage track, a per-PR **CheckGantt** (with flake-radar and
duration-regression badges), a cost chip, a **"ready + auto-merge"** action, and — for
merged PRs — a **lead-time Waterfall** ("where did the time go").

### Intended & possible use cases

- `[DEV]` Filter to just my PRs and see each one's stage + ETA + "ball in my court" badge
  in one place, so I stop polling.
- `[DEV]` See which single check is the long pole on my PR so I optimize the right thing
  (or split the PR) instead of guessing.
- `[DEV/RM]` Get a "ready + auto-merge" button the moment a PR is mergeable, so it merges
  itself and I walk away.
- `[RM]` Read p50/p90 queue ETAs off the metro train to give a credible merge time to
  engineers and stakeholders.
- `[RM]` Filter to the "awaiting-prod" cohort and batch a single prod promotion instead
  of N risky one-off deploys.
- `[RM]` Reorder/prioritize a hotfix to the front of the queue ahead of routine feature
  PRs.
- `[SRE]` Trip a failed-count delta badge on a spike over a rolling window so a sudden
  systemic break is distinct from steady background failure.
- `[SRE]` Read deploy reachability per env (online/unreachable + commit SHA) to tell
  "deploy stuck" from "deployed but env is down".
- `[SRE]` See how long a stage has been stuck vs its normal p50/p95 so "slow" and "hung"
  look different and I escalate only real hangs.
- `[PLAT]` Surface concurrency-group contention (jobs waiting on a mutex) on the Gantt to
  right-size limits that silently serialize work.
- `[PLAT/RM]` Spot the PR idle/stuck-at-one-stage longest, to chase the owner before it
  ages out of the train.
- `[DEV/SRE]` Freeze a merged-PR lead-time Waterfall as a shareable artifact — for
  setting expectations or for an incident retro.
- `[EM]` Aggregate "work-in-flight" and total queue depth across all repos, and the
  lead-time waterfall as median/p90, to see systemic congestion and which stage eats the
  most calendar time.
- `[EM]` Filter by team / repo-tier and compare queue-wait vs a trailing 4-week baseline
  to benchmark squads.

**Home base for:** PR Author (where I live while waiting) and Release Manager during an
active merge window.

## 4. Diagnose  *(mode: drill)*

### What it is
"Why is this PR stuck?" A two-column drill surface: a **PR list** (focused repo pinned
first) on the left; a **detail panel** on the right showing the PR's single **blocker**
in active voice ("X failed and is blocking this PR" / "X is failing — likely a flake" /
"Waiting on X" / "All checks passed") plus a **CheckGantt**. Above it, three conditional
banners: **systemic failure clusters** (same check failing across 3+ PRs/repos), **queue
incidents** (stalled-queue recovery steps), and **flake remediation proposals**
(quarantine candidates with rationale and blast radius).

### Intended & possible use cases

- `[DEV]` Pick my PR and jump straight to the exact failing check + log excerpt, instead
  of scrolling raw CI output.
- `[DEV/SRE]` See a flake-likelihood annotation on the failing check so I instantly know
  whether to re-run or actually fix something.
- `[DEV]` One-click "re-run just this check" when flake-likelihood is high, without
  burning the whole suite.
- `[PLAT/SRE]` Have systemic failure clusters grouped automatically so I fix one root
  cause instead of triaging N identical red PRs.
- `[SRE/PLAT]` Get an ordered **recovery runbook** (not a hint) for a stalled queue, so
  any on-call — even unfamiliar with the repo — can recover confidently.
- `[PLAT/EM/SRE]` Show each cluster's **blast radius** (PRs × repos × engineer-hours ×
  $) so I prioritize the break costing the most throughput.
- `[SRE]` Show a "first seen / introducing commit or workflow change" pointer on each
  cluster so I jump to the likely cause instead of bisecting by hand.
- `[PLAT]` Distinguish infra-caused failures (runner OOM, registry timeout) from
  code-caused ones so I route the fix to the right owner.
- `[RM]` Produce a copy-ready status summary (blocker, owner, ETA, next step) to paste
  into the incident channel or author thread.
- `[SRE]` Mark a cluster acknowledged / mitigating / resolved so co-on-call doesn't
  double-respond.
- `[EM]` Track recurring clusters over time (chronic vs one-off) and attach an
  effort/savings estimate, to fund the highest-payoff fix as a portfolio decision.
- `[DEV/PLAT]` Surface the flake-remediation proposal inline with a one-click apply, so
  fixing it unblocks me *and* everyone else.

**Home base for:** the incident cockpit — SRE during a fire, Platform during a flake
storm.

## 5. Model & Edit  *(mode: act)*

A three-tab surface that consolidates the legacy Model, Optimize, and Build views. Every
change is server-validated; **no required gate can be silently dropped.**

### 5a. Inspect
**What it is.** A read-only **protection matrix** (rows = checks, columns = tiers; cells
show gate/conditional/advisory/absent glyphs + drift ⚠). Toggleable heat overlays for
**Cost** and **Quality**, a required-gates summary chip, **ruleset-sync** status
(model vs GitHub branch protection), a **security findings** list (e.g.
broad-permissions), and a per-cell **drill-down drawer** with provenance, per-tier
evidence (runs / fail% / minutes), a what-if simulator, and a "Copy Prompt" button.

- `[RM/PLAT]` Confirm prod is actually protected by the gates I think it is — see which
  checks gate which tier.
- `[PLAT/RM]` Catch **drift**: a repo silently missing (or gaining) a required gate, or a
  model out of sync with branch protection.
- `[PLAT/SRE]` Flag **security findings** (over-broad workflow permissions) inline so I
  close supply-chain holes before they're exploited.
- `[PLAT/EM]` Read cost/quality **heat** per cell (and rolled up per team/repo) to spot
  the most expensive, lowest-value checks.
- `[DEV]` See which red checks actually block merge vs which are advisory, so I know what
  I can safely ignore.
- `[PLAT]` Open a cell's evidence drawer to see provenance + per-tier runs/fail%/minutes
  before deciding anything.

### 5b. Optimize
**What it is.** Cost-driven **demotion candidates** ("never failed — demote candidate"),
per-check **Simulate demote** and **Quarantine (flaky)** actions with safety verdicts, a
**required-check-prefix lever** (preview + draft PR), a multi-change **plan** simulator,
**calibration accuracy** ("X% accurate over M landed changes"), and draft-PR / Claude
prompt generation.

- `[PLAT]` Rank demotion candidates by cost and move expensive low-signal checks to a
  cheaper tier (PR → nightly), biggest savings first.
- `[PLAT/RM]` Simulate a tier move and get a **safety verdict** before applying, so I
  never weaken a required gate or expose the default branch by accident.
- `[SRE/PLAT/RM]` Quarantine a flaky gate in one or two clicks to stop it killing the
  queue immediately, while the real fix follows.
- `[PLAT]` See **calibration accuracy** next to every estimate so I know how much to
  trust the projected savings.
- `[EM]` See projected monthly $ saved + a risk verdict per candidate, and a "draft the
  PRs for these N optimizations" batch action so approved savings actually land.
- `[DEV]` Spot demotion candidates (slow checks that never catch anything) to advocate
  moving them off the critical path slowing my PR.

### 5c. Edit
**What it is.** A visual **DAG pipeline canvas** (lanes per tier, nodes per check, edges
for `needs`). Select a node to apply structured mutations — **timeout / shift-left /
remove / runner-class / concurrency** — building a reversible **mutation stack** with a
live **safety verdict** (safe / blocked-drops-required-gate / low-confidence-scaffold).
Produces a generated diff + draft PR, with a **raw-YAML escape hatch** that runs the same
verdict.

- `[PLAT]` Apply timeout/shift-left/runner/concurrency mutations and get a generated PR
  (or Claude prompt) so every pipeline change is reviewable and reversible — never a
  console click.
- `[PLAT/RM/DEV]` Have the editor **block** dropping or weakening a required gate, so a
  well-intentioned edit can't open a quality or security hole.
- `[SRE]` Make an **emergency** pipeline edit under pressure with a pre-apply safety
  verdict and a one-click revert.
- `[DEV]` Learn the pipeline shape visually (which stages feed which) without reading a
  line of YAML.
- `[PLAT/SRE]` Keep a raw-YAML escape hatch *with the same guardrails* for edge cases the
  visual editor can't model.
- `[SRE]` Require a reason on every quarantine/edit and auto-log it to the changelog, so
  the emergency change is auditable and reversible in the retro.

**Home base for:** Platform / DevEx Engineer — this is where safe pipeline *evolution*
happens (inspect drift/security → decide the move → simulate with a verdict → emit a
reviewable PR).

## 6. Insights  *(mode: read)*

### What it is
The analytics home, two tabs. **Analytics** (lazy-loaded) groups ~25 panels into five
sub-sections selectable by a button group:

- **Tuning** — ranked recommendations (filter by priority, "view evidence →" deep-links)
  and a recent-config-changes feed (overlaid as markers on charts).
- **Throughput & Queue** — PR-count trends, queue throughput + time-in-queue p50,
  queue efficiency, a **batch-size advisor** (queueing-theory what-if), merge velocity +
  deploy lag, live duration regressions.
- **Performance** — runner-wait health, runner pools + starvation alerts, concurrency
  demand, slowest/most-variable jobs, **critical path** + slack, an interactive
  **CI needs graph** (DAG), ETA calibration.
- **Reliability** — flakiest jobs, demotion/promotion candidates (with Draft PR + Claude
  prompt), spot reclaims, **train-killers**, workflow lint (timeout calibration), runner
  routing.
- **Cost** — CI cost auto-rate, cost actuals + attribution coverage, per-repo / per-pool
  / per-job / per-run cost, **DORA-lite lead-time** waterfall.

The **Tuning & Policy** tab adds **budgets** gauges, **policy** violations, closed-loop
**outcomes** (did applied changes actually help?), and a **changelog & audit** trail.
All charts support 24h–30d windows with hour/day buckets; every stat carries a
definition tooltip mirrored from the Legend.

### Intended & possible use cases

- `[EM]` Trend the four DORA metrics over 30/60/90 days with per-team breakdowns to
  benchmark squads and show trajectory upward.
- `[EM/PLAT]` Attribute cost actuals per team / repo / job and as **$/merged-PR**, for
  grounded efficiency and chargeback conversations.
- `[EM]` Read budget gauges with projected end-of-period spend to course-correct
  mid-cycle instead of explaining an overrun after the fact.
- `[PLAT/EM]` Track **closed-loop outcomes** (before/after on each shipped change) to
  prove platform-investment ROI — or roll a change back if it didn't help.
- `[PLAT/SRE]` Watch runner-wait health + pool **starvation** alerts to right-size pools
  and kill the wait that inflates every PR's lead time.
- `[SRE]` See **live duration-regression** detection to catch a job that doubled in
  runtime *today* before it silently eats the queue.
- `[PLAT/SRE]` Rank **train-killers** (checks that most often eject the merge train) and
  read spot-reclaim/concurrency-peak timelines to drive durable fixes and correlate
  stalls with infra pressure.
- `[PLAT]` Use the batch-size advisor + concurrency curves to tune merge-queue batching
  and concurrency caps to real demand.
- `[RM/PLAT/DEV]` Read **ETA calibration** to know whether the p50/p90 promises made on
  Pipeline are trustworthy (and catch model drift).
- `[PLAT/SRE]` Run **workflow lint** (missing/oversized timeouts) to prevent the next
  infinite-hang incident proactively.
- `[EM]` Export any view to CSV/PDF or a scheduled report so the QBR/board deck builds
  itself from source data.
- `[EM/SRE]` Filter the **audit changelog** by who/what/when for governance and to answer
  "what changed right before this broke."
- `[DEV]` Read flake-candidate history to back an argument that a specific check should be
  quarantined.

**Home base for:** Engineering Manager (aggregates, trends, money, DORA, ROI) and the
quiet backbone that keeps the action-screens honest.

## 7. Shell & Overlays  *(plus alternate delivery modes)*

### What it is
The chrome and global surfaces: a header **spine** (brand, **Pipeline Switcher**,
**liveness** indicator, **self-health dot**, **⌘K**, legend, notification bell,
settings), a left **rail** of the five sections, and several overlays —
**Command Palette** (jump to section / repo / PR), **Settings Panel** (watched repos,
tuning knobs, intervals, notifications, restart), **Legend Panel** (the visual-vocabulary
glossary), **Protection Drawer** (per-check evidence + what-if), **Forecast Banner**, and
**Status Strip**. Plus three delivery modes: **standalone classic app**, **embeddable
component** (host owns auth/chrome; controlled or uncontrolled repo focus), and **kiosk**
(auto-cycling, read-only wall display).

### Intended & possible use cases

- `[RM/PLAT/SRE]` A **command palette** to jump to any section / repo / PR / runbook by
  typing, so I navigate at incident speed without clicking through tabs.
- `[DEV]` Jump straight to my PR by number or branch from the palette and land on exactly
  what I'm waiting for.
- `[SRE/RM/EM]` A **kiosk wall-display** that auto-cycles key views for the team room, so
  current delivery state is always radiating with no one driving — and **auto-pins** to
  the active incident view when one fires.
- `[DEV/SRE/RM]` **Notifications** tuned to fire only on signals that matter (my PR
  green/merged/broke; main red; queue wedged; deploy stuck; pool starved) so the pager
  means something and I'm not desensitized.
- `[EM/SRE]` A periodic **digest** (email/Slack, batched by severity) so slow-burn issues
  reach me async while fire drills page me.
- `[PLAT/EM]` **Settings** for watched repos, sweep/hot/deploy intervals, budget
  thresholds, and per-signal notification trip-points, so polling cost and alert noise
  match my team's tolerance.
- `[SRE/PLAT]` A **self-health/liveness** state surfaced in overlays and kiosk, so a
  stale dashboard on the wall announces itself rather than lying quietly.
- `[EM/PLAT/SRE]` **Embeddable** host mode so the live spine drops into our existing
  admin/incident portal under one auth, alongside other ops tooling.
- `[DEV/EM]` A **legend/glossary** always reachable so anyone handed the screen can
  self-serve "what does amber mean."
- `[EM]` **Saved views** per audience (my staff / my director / finance) so each
  stakeholder sees the cut that matters.
- `[DEV]` Scope the whole dashboard to "my PRs" by default so every screen opens
  pre-filtered to what I care about.

## 8. Cross-screen workflows

Jobs-to-be-done that span multiple screens — the connective tissue the personas keep
reaching for:

1. **Cost spike → fix → proof** `[PLAT/EM]`
   Health cost-forecast banner goes red → **Insights** cost attribution finds the driver
   workflow/job and $/merged-PR → **Model & Edit › Optimize** ranks it as a demotion
   candidate, simulate → safety verdict → draft PR → **Insights › Outcomes** confirms the
   spend curve actually bent (or roll back).

2. **Flake storm → quarantine → real fix → verify** `[PLAT/SRE/EM]`
   Health flakiest leaderboard spikes → **Diagnose** clusters it as one systemic check
   with blast radius (introducing commit shown) → one-click **quarantine** draft PR
   unblocks the fleet → file the real-fix follow-up → **Insights** flakiest-jobs +
   closed-loop verify the flake rate fell and dev-hours recovered.

3. **Gate drift / security hole → safe correction** `[PLAT/SRE]`
   **Model & Edit › Inspect** drift detection flags a required check that vanished (or a
   broad-permissions finding) → ruleset-sync confirms mismatch vs branch protection →
   Edit/Optimize simulates the correction with a verdict that **blocks any weakening** →
   draft PR restores the gate → **Insights › Audit** records it and confirms coverage is
   back to 100%.

4. **Merge queue wedges** `[RM/SRE]`
   Health "queue blocked" verdict → **Pipeline** metro-train shows the head PR stalling
   everyone → **Diagnose** queue-incident runbook gives ordered recovery steps → mitigate
   (eject/hold head PR, quarantine the train-killer) → back to **Pipeline** to confirm
   flow resumed; **Insights** train-killer ranking confirms whether it's a repeat
   offender worth a durable fix.

5. **Fire-and-forget merge** `[DEV]`
   **Pipeline** "ready + auto-merge" → **Shell** browser notification on
   green/merge/break. Set once, get pinged only on a state change, stop polling entirely.

6. **Lead-time regression → bottleneck → benchmark → report** `[EM]`
   **Pipeline** lead-time waterfall flags a ballooning stage fleet-wide → **Insights**
   DORA lead-time + critical path + per-team breakdown localizes the squad/stage →
   **Model & Edit** simulates a relieving change → digest/export the before/after upward.

## 9. Observations & opportunities

A few cross-cutting themes surfaced by the persona brainstorm — candidate gaps worth
deliberate product decisions, *not* implementation claims:

- **Personalization / "my PRs" scoping.** Multiple personas (esp. DEV) want every screen
  scoped to a person, team, or repo by default. Today focus is repo-centric; a "who"
  axis (team/owner) recurs across Pipeline, Diagnose, Insights, and notifications.
- **Acknowledge / ownership state on incidents.** SRE and EM both want clusters/incidents
  to be *assignable* and to carry ack/mitigating/resolved state — turning a wall of red
  into accountability. The current Diagnose banners are read-only presentations.
- **Trend/delta everywhere.** "Is this a degrading green?" appears for Health lanes,
  Insights panels, and Pipeline queue-wait. A consistent delta-vs-baseline treatment
  would serve monitor, read, and report jobs at once.
- **Blast-radius quantification.** Diagnose clusters and flake proposals are far more
  actionable when expressed in PRs × repos × engineer-hours × dollars — the unit that
  lets EM fund and RM/SRE prioritize.
- **Reporting & export.** EM specifically wants snapshots, CSV/PDF export, scheduled
  digests, and saved per-audience views — a "communicate upward" capability that spans
  Health, Insights, and the shell.
- **Closed-loop as the spine of trust.** Action screens (Model & Edit) are only as
  credible as Insights' calibration and outcome tracking. Keeping that loop tight is what
  lets every persona trust the ETAs, savings estimates, and safety verdicts.
